### 3.3 存储格式

#### 方式A：CSV格式（必做）

所有原始数据和清洗后的数据均以CSV格式存储。

**优点**：
- 通用性强，几乎所有数据分析工具都支持CSV读取
- 可直接用文本编辑器查看，便于调试和验证
- 无需额外依赖库

**缺点**：
- 文件体积较大，对大型数据集不友好
- 读取速度较慢，特别是频繁读取时
- 不保留数据类型信息（如datetime需要重新转换）

**CSV 格式会力不从心的场景**：
- 数据量达到 GB 级别时，读取速度显著下降，内存占用高
- 需要频繁读取部分列时，CSV 必须加载全部数据再筛选，效率低下
- 数据包含复杂类型（如嵌套结构、数组）时，CSV 无法直接表示
- 大规模并发读取场景下，I/O 瓶颈明显
- 需要保留精确数据类型时，每次读取都需重新推断和转换类型

#### 方式B：Parquet格式（进阶，已实现）

将合并后的数据同时保存为Parquet格式。

**优点**：
- 列式存储，读取特定列时效率更高
- 文件体积小，压缩效率高（比CSV小5-10倍）
- 保留Schema信息，数据类型无需手动转换
- 支持分区存储，便于大数据处理

**缺点**：
- 需要额外依赖（pyarrow或fastparquet）
- 不能直接用文本编辑器查看
- 跨平台兼容性不如CSV

#### 方式C：SQLite数据库（进阶，已实现）

设计3张表存储数据，演示跨表JOIN查询。

**优点**：
- 单文件数据库，便于管理和备份
- 支持SQL查询，适合复杂的多表关联分析
- 数据完整性约束（主键、外键等）

**缺点**：
- 需要SQL知识
- 大批量写入时性能不如CSV/Parquet直接存储

### 3.4 进阶格式选择理由

本作业同时实现了方式B（Parquet）和方式C（SQLite）两种进阶格式，理由如下：

1. **Parquet 选择理由**：本次作业数据规模较小（约3.8MB），Parquet的优势尚不明显。但掌握 Parquet 对未来处理大规模金融数据（如全市场日度行情、分钟级数据）非常重要。Parquet 的列式存储特性使其在读取部分列时效率远高于 CSV，且文件体积仅为 CSV 的 20% 左右。

2. **SQLite 选择理由**：金融数据分析经常需要多表关联查询（如股票行情与财务数据、宏观数据的 JOIN）。SQLite 提供了标准 SQL 接口，便于演示复杂查询场景，同时单文件特性便于备份和迁移。

3. **同时实现的价值**：两种格式服务于不同场景。Parquet 适合大规模数据的高效读写，SQLite 适合复杂的多表关联分析。掌握这两种工具能够应对不同的实际需求。

```bash

# 安装依赖
```

```bash

jupyter notebook 03_analysis.ipynb
```
